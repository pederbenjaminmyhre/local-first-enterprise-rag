Lessons Learned

This document captures key insights gained during the development and deployment of the Local-First Enterprise RAG system. These lessons are grouped by theme and presented in order of impact.

Infrastructure & Networking

Infrastructure

IPv6 Connectivity Can Silently Break Docker and Azure CLI

Docker Desktop on Windows (via WSL2) and the Azure CLI both resolve hostnames to IPv6 by default. When IPv6 connectivity is unreliable — which is common on residential networks and some corporate firewalls — Docker image pulls and Azure deployments fail with cryptic "Connection reset" errors. The fix required adding IPv4 entries to the WSL2 VM's hosts file and temporarily disabling IPv6 on the Wi-Fi adapter. Takeaway: When troubleshooting connection failures in containerized environments, always check IPv4 vs. IPv6 resolution first.

Infrastructure

Azure Resource Provider Registration Is Not Automatic

A fresh Azure subscription requires manual registration of resource providers (Microsoft.Compute, Microsoft.Web, Microsoft.App, Microsoft.Sql, Microsoft.OperationalInsights) before any resources can be deployed. This is not obvious from error messages, which simply report "validation failed." Additionally, VM SKU quotas (Free, Basic, Standard) are separate per-SKU limits, distinct from the general "Total Regional vCPUs" quota. Takeaway: Register all required providers and request SKU-specific quotas before attempting Bicep deployments on new subscriptions.

Infrastructure

SQL Server 2025 Docker Tag Naming Is Inconsistent

The tag 2025-latest exists on Microsoft Container Registry but was intermittently unreachable. The specific cumulative update tag (2025-CU3-ubuntu-22.04) also failed due to the same IPv6 issue. SQL Server 2025 has reached GA and is no longer "preview," but documentation still references preview tags. Takeaway: Always verify Docker image tags against the actual MCR registry, and have fallback tags ready.

Performance & Model Management

Performance

Ollama Model Swapping Is the Primary Bottleneck

Ollama loads only one model into memory at a time. Switching between the embedding model (mxbai-embed-large, 670 MB) and the generation model (llama3.2, 2 GB) takes 30–120 seconds depending on hardware and whether the container is using a volume mount or in-memory storage. This dominated test execution time and required test ordering strategies. Takeaway: Design test suites and application workflows to minimize model swaps — group all embedding operations before generation operations.

Performance

CPU-Only LLM Inference Is Viable but Slow

On an Intel Iris Xe (integrated GPU, no CUDA), llama3.2:1b generates ~200 characters in 20–25 seconds. The full llama3.2 (3.2B params) takes over 60 seconds. This is acceptable for a demo but not for production. The same queries run in 2–3 seconds with a dedicated GPU (RTX 3060 or better). Takeaway: Always communicate hardware requirements alongside performance claims. The architecture is production-ready; the latency is a hardware constraint.

Testing

Testing

FluentAssertions v8.x Hangs the xUnit Test Runner

FluentAssertions 8.x introduced a commercial license check that, in certain environments, causes the test host process to hang indefinitely after printing the license warning. Downgrading to v7.2.0 resolved the issue immediately. The unit tests (which don't trigger the license check timing) were unaffected, making this difficult to diagnose. Takeaway: Pin test library versions and be cautious of major version upgrades that change licensing models.

Testing

Synchronous HTTP Calls Deadlock in xUnit

The initial SkippableFact implementation used .GetAwaiter().GetResult() for a synchronous Ollama health check. This deadlocked within xUnit's synchronization context. Replacing it with IAsyncLifetime.InitializeAsync and standard [Fact] attributes resolved the issue. Takeaway: Never use blocking async calls (.Result, .GetAwaiter().GetResult()) in test frameworks that manage their own synchronization context.

Cloud Deployment

Cloud

Azure SQL Database Supports VECTOR but Not .bak Restore

Azure SQL Database (Basic tier) supports the VECTOR(1024) data type and VECTOR_DISTANCE function, confirming compatibility with the local SQL Server 2025 vector queries. However, it does not support restoring .bak backup files — data must be seeded via T-SQL scripts or BACPAC import. A seed script with 25 representative products was created for the Azure demo. Takeaway: Maintain both a backup-based approach (Docker/on-prem) and a script-based approach (cloud) for database initialization.

Cloud

Container App Models Don't Persist Across Restarts

Ollama running in Azure Container Apps stores pulled models in the container's ephemeral filesystem. When the container scales to zero and restarts, models must be re-pulled. For production, a persistent volume or a custom Docker image with pre-baked models would be required. Takeaway: Scale-to-zero saves cost but introduces cold-start latency for both container boot and model loading.

Cloud

Region Selection Matters for New Subscriptions

East US refused SQL Server creation ("not accepting new servers") while Central US worked. Quota availability and region restrictions vary by subscription type and age. Takeaway: Have 2–3 fallback regions planned, and create resources incrementally via CLI rather than all-at-once via Bicep when debugging deployment issues.
